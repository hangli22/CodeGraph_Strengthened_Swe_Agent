\section{背景与相关工作}

\subsection{面向软件工程的大语言模型 Agent}

近年来，大语言模型在代码生成、代码理解和自动程序修复等任务中取得了显著进展。随着模型能力的提升，研究者开始将大语言模型从单轮代码生成扩展到真实软件工程任务中，使其能够根据 issue 描述定位缺陷、修改代码、运行测试并生成补丁。SWE-bench 是该方向中具有代表性的评测基准之一，其任务来自真实 GitHub issue 和对应 pull request，要求模型在给定完整代码仓库和问题描述的条件下生成能够通过测试的补丁 \cite{swebench}。与传统函数级或文件级代码生成任务相比，SWE-bench 更强调仓库级上下文理解、跨文件定位、环境交互和多步推理能力，因此被广泛用于评估大语言模型在真实软件维护场景中的能力。

在这一背景下，基于工具调用的软件工程 Agent 成为重要研究方向。SWE-agent 通过设计 Agent-Computer Interface，使语言模型能够使用命令行、查看文件、编辑代码并运行测试，从而在真实软件仓库中执行端到端修复任务 \cite{sweagent}。该工作表明，工具接口设计会显著影响 Agent 在代码导航、文件编辑和测试执行中的表现。OpenHands 等通用软件工程 Agent 进一步探索了语言模型在软件开发环境中的自主操作能力，使 Agent 能够在沙箱环境中完成代码修改、命令执行和任务求解等操作 \cite{openhands}。这类工作推动了软件工程任务从“模型直接生成答案”向“模型与开发环境交互”的范式转变。

然而，现有软件工程 Agent 在仓库级代码理解方面仍存在不足。SWE-agent 等方法虽然提供了较强的环境交互能力，但 Agent 对仓库结构的理解主要依赖命令行搜索、文件浏览和自身推理。当缺陷涉及多个文件、多个类或需要分析函数调用、类继承和方法重写等结构关系时，Agent 容易出现搜索范围过大、重复查看无关文件、遗漏关键代码位置以及后期难以收敛等问题。因此，仅为模型提供底层环境操作接口并不足以解决真实仓库中的代码定位和上下文组织问题。

与 Agent 式方法不同，Agentless 采用更固定的自动修复流程，将软件工程任务拆分为定位、修复和补丁验证等阶段，避免让模型在复杂工具空间中自主规划下一步动作 \cite{agentless}。该方法在 SWE-bench Lite 上取得了较强表现，说明在真实软件修复任务中，缺陷定位和流程控制是影响最终修复效果的重要因素。Agentless 的成功也表明，复杂的自主 Agent 并非解决软件修复任务的唯一方式，简化且可控的流程同样具有竞争力。然而，固定流程方法在一定程度上牺牲了 Agent 的交互灵活性，难以充分利用多轮探索过程中产生的中间观察结果。因此，如何在保留 Agent 灵活性的同时增强其仓库级代码定位能力，仍然是一个值得研究的问题。

\subsection{仓库级代码检索}

代码检索是软件工程 Agent 理解仓库的重要基础。传统信息检索方法如 BM25 能够根据词项匹配程度衡量查询与文档之间的相关性，适合处理函数名、类名、变量名、异常信息、错误消息和文件路径等精确符号 \cite{bm25}。在软件缺陷修复场景中，issue 描述常常包含这些符号级线索，例如报错堆栈、配置项名称、API 名称或测试失败信息。因此，关键词检索在 issue-to-code 定位中仍具有重要价值。

随着神经网络表示学习的发展，语义向量检索逐渐成为代码检索的重要方法。CodeBERT 等预训练模型通过联合建模自然语言和程序语言，支持自然语言代码搜索、代码文档生成等任务 \cite{codebert}。Dense Passage Retrieval 等工作表明，使用双编码器学习查询和文档的稠密向量表示，可以在开放域问答等任务中有效替代或补充传统稀疏检索 \cite{dpr}。在大语言模型应用中，Retrieval-Augmented Generation 将外部知识检索与生成模型结合，使模型能够利用非参数化记忆缓解知识不足和上下文受限问题 \cite{rag}。这些思想也被广泛迁移到代码智能任务中，用于为模型提供与当前问题相关的代码上下文。

然而，真实软件仓库中的代码检索与普通文本检索存在显著差异。首先，代码中的标识符、路径、函数签名和错误信息具有强符号性，单纯语义检索可能忽略关键字符串匹配。其次，issue 中的自然语言描述与代码实现之间可能存在语义鸿沟，单纯关键词检索又可能无法召回行为相关但字面不匹配的代码。最后，代码实体之间存在大量结构关系，例如包含、导入、调用、继承和重写等，仅依赖文本相似度难以发现与入口节点结构相关但文本不相似的关键位置。因此，仓库级软件修复通常需要同时利用关键词匹配、语义相似度和程序结构关系。

已有仓库级代码生成和补全研究也注意到了这一问题。GraphCoder 面向仓库级代码补全任务，提出基于代码上下文图的检索增强框架，通过控制流、数据依赖和控制依赖等结构信息捕获补全目标附近的上下文，并采用 coarse-to-fine 检索过程定位相似代码片段 \cite{graphcoder}。CodeRAG 面向真实仓库级代码生成，构建需求图和 dependency-semantic code graph，通过图上的支持代码检索帮助模型生成能够融入当前仓库的代码 \cite{coderag}。这些研究表明，仓库级任务中的检索不能仅停留在文本片段层面，而需要结合代码结构、依赖关系和任务需求进行上下文构造。

\subsection{面向仓库理解的代码图方法}

代码图通过节点和边表示程序实体及其关系，是仓库级代码理解的重要工具。常见代码图可以表示文件、模块、类、函数、方法、语句等不同粒度的代码实体，也可以表示包含、导入、调用、继承、数据依赖和控制依赖等关系。与普通文本索引相比，代码图能够显式建模程序结构，使系统不仅能够找到与查询文本相似的代码，还能够进一步发现与候选节点存在结构联系的相关代码区域。

RepoGraph 是近期面向 AI 软件工程任务的代表性代码图方法之一。该工作提出将仓库级结构作为可插拔模块，为 AI software engineering systems 提供 repository-wide navigation，并在 SWE-bench 等任务中验证其对不同系统的增强作用 \cite{repograph}。RepoGraph 的核心思想是，现代软件工程任务不仅要求模型具备代码生成能力，还要求其能够管理和交互整个代码仓库；因此，仓库级结构理解对于把握更大范围上下文和生成有效修改十分关键。与仅依赖文件级浏览的方法相比，RepoGraph 强调利用仓库范围内的结构信息辅助模型定位相关代码。

除了 RepoGraph，知识图增强的软件修复也成为新的研究方向。KGCompass 面向 repository-level software repair，构建 repository-aware knowledge graph，将 issue、pull request 和代码实体联系起来，并利用图路径缩小搜索空间、增强补丁生成 \cite{kgcompass}。该方法在 SWE-bench Lite 上报告了较强的修复性能和函数级定位能力，并指出许多缺陷定位需要通过知识图中的多跳关系完成。这说明，在真实缺陷修复任务中，相关代码往往并不直接出现在 issue 描述中，而需要通过调用、依赖或其他结构路径间接发现。

图增强方法也被用于仓库级代码生成 Agent。CodeRAG 和 GraphCodeAgent 等工作通过需求图和代码图建立自然语言需求与仓库实现之间的连接，引导模型检索支持代码并完成复杂依赖场景下的代码生成 \cite{coderag, graphcodeagent}。这些方法与本文关注的问题具有相似动机，即真实软件工程任务中的目标代码通常依赖仓库内已有实现、接口约定和跨文件关系，模型需要结构化检索机制才能有效利用这些信息。不过，这类工作主要面向仓库级代码生成或补全，而不是直接面向 SWE-bench 式缺陷修复 Agent 的多轮修复轨迹。

总体来看，代码图增强方法已经证明了结构信息在仓库级软件工程任务中的价值。然而，现有方法仍存在若干不足。第一，部分方法强调一次性仓库结构检索或固定流程修复，而没有充分考虑 Agent 在多轮交互中不断提出新查询、逐步深化局部代码区域的需求。第二，完整构建细粒度代码图通常带来较高成本，尤其在 SWE-bench 这类包含大量真实仓库实例的评测中，预先构建完整 method 级图会增加解析、存储和 embedding 开销。第三，现有检索结果常以代码片段、候选文件或子图形式返回，缺少面向 Agent 决策的结构化解释信息，例如候选节点为何相关、与其他节点存在何种关系、是否值得进一步查看或深化等。

\subsection{研究空白与本文定位}

综上，现有研究可以大致分为三类。第一类是 SWE-agent、OpenHands 等软件工程 Agent，它们强调模型与开发环境的交互能力，但对仓库级结构化检索支持不足 \cite{sweagent, openhands}。第二类是 Agentless 等固定流程方法，它们通过定位、修复和验证等阶段降低自主 Agent 的复杂性，但相对缺少多轮交互中的动态检索能力 \cite{agentless}。第三类是 RepoGraph、GraphCoder、CodeRAG 和 KGCompass 等代码图或图增强检索方法，它们证明了仓库结构对于代码补全、代码生成和软件修复的重要性，但多数工作并未专门围绕轻量软件修复 Agent 的多轮交互过程设计可逐步深化的检索机制 \cite{repograph, graphcoder, coderag, kgcompass}。

本文工作位于上述研究方向的交叉处。本文并不试图从零设计新的软件工程 Agent，也不直接改造大语言模型本体，而是在 mini-swe-agent 这类轻量 Agent 框架中引入代码图增强检索能力。与原始 SWE-agent 主要关注 Agent-Computer Interface 不同，本文更关注如何为 Agent 提供结构化、轻量化、可交互的仓库级代码上下文。与固定流程方法不同，本文保留 Agent 的多轮工具调用能力，使其能够根据中间结果动态执行检索和深化。与已有代码图方法相比，本文进一步提出两阶段代码图表示：首先构建轻量骨架图支持全仓库范围的快速定位，然后在 Agent 需要细粒度信息时，对候选文件或相关结构邻域执行按需深化，补充 method 级上下文。
