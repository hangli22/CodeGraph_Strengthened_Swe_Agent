\section{背景与相关工作}

\subsection{面向软件工程任务的大语言模型 Agent}

近年来，大语言模型逐渐从单轮代码生成扩展到真实软件工程任务中，例如根据 issue 描述定位缺陷、修改代码、运行测试并生成补丁。SWE-bench 是该方向中具有代表性的评测基准，其任务来自真实 GitHub issue 和对应 pull request，要求模型在给定完整代码仓库和问题描述的条件下生成能够通过测试的补丁 \cite{swebench}。与传统函数级或文件级代码生成任务相比，SWE-bench 更强调仓库级上下文理解、跨文件定位、环境交互和多步推理能力，因此成为评估大语言模型软件工程能力的重要基准。

SWE-agent 进一步推动了基于工具调用的自动软件修复研究。该工作提出 Agent-Computer Interface，使语言模型能够使用命令行、浏览文件、编辑代码并运行测试，从而在真实软件仓库中执行端到端修复任务 \cite{sweagent}。SWE-agent 的核心贡献在于说明接口设计会显著影响模型在仓库导航、代码编辑和测试执行中的表现。此后，许多软件修复和 issue resolving 研究都以 SWE-bench 和 SWE-agent 类框架为基础，继续探索如何改进代码定位、上下文构造、补丁生成和验证流程。

然而，SWE-agent 类方法虽然提供了较强的环境交互能力，但 Agent 对仓库结构的理解仍然主要依赖命令行搜索、文件浏览和自身推理。当缺陷涉及多个文件、多个类，或需要分析函数调用、类继承、方法重写等结构关系时，Agent 容易出现搜索范围过大、重复查看无关文件、遗漏关键代码位置以及后期难以收敛等问题。因此，仅提供底层操作接口并不足以解决真实仓库中的代码定位和上下文组织问题，仍需要更有效的仓库级检索与结构化上下文机制。

与自主 Agent 路线不同，Agentless 采用更固定的自动修复流程，将任务拆分为定位、修复和补丁验证等阶段，避免让模型在复杂工具空间中自主规划下一步动作 \cite{agentless}。该方法表明，在真实软件修复任务中，缺陷定位和流程控制对最终修复效果具有重要影响。不过，固定流程方法也减少了 Agent 根据中间观察结果动态调整检索目标和修复策略的空间。因此，一个更合适的问题是：如何在保留 Agent 多轮交互能力的同时，为其提供更稳定、更结构化的仓库级代码定位能力。

RepairAgent 和 AutoCodeRover 也从自动程序修复角度探索了 LLM Agent 的工具使用能力。RepairAgent 将大语言模型作为能够自主调用工具的修复 Agent，使其在缺陷分析、补丁生成和测试反馈之间迭代 \cite{repairagent}。AutoCodeRover 则更强调面向软件工程的程序结构搜索，将抽象语法树、类和方法等结构信息用于 issue 相关上下文定位，并在 SWE-bench Lite 上验证了结构化搜索对修复任务的价值 \cite{autocoderover}。这些工作说明，真实软件修复不仅需要模型生成代码，更需要有效的仓库理解和代码定位机制。

\subsection{仓库级代码检索}

代码检索是软件修复 Agent 理解仓库的重要基础。传统信息检索方法如 BM25 能够根据词项匹配程度衡量查询与文档之间的相关性，适合处理函数名、类名、变量名、异常信息、错误消息和文件路径等精确符号 \cite{bm25}。在软件缺陷修复场景中，issue 描述和 Agent 后续查询经常包含这些符号级线索，例如报错堆栈、配置项名称、API 名称、测试名称或函数名。因此，关键词检索在 issue-to-code 定位中仍具有重要价值。

语义检索则通过向量表示捕获自然语言查询与代码片段之间的语义相关性。CodeBERT、GraphCodeBERT、UniXcoder、CodeT5 和 CodeT5+ 等预训练代码模型为自然语言代码搜索、代码摘要和代码生成等任务提供了基础表示能力 \cite{codebert, graphcodebert, unixcoder, codet5, codet5plus}。在更一般的检索增强生成研究中，DPR 和 RAG 表明，稠密检索与生成模型结合能够为模型提供外部知识，从而缓解上下文不足和知识缺失问题 \cite{dpr, rag}。这些思想被迁移到代码任务后，形成了面向代码生成、代码补全和代码修复的检索增强方法。

不过，真实软件仓库中的代码检索不能只依赖单一检索信号。一方面，代码中的标识符、路径和错误信息具有强符号性，单纯语义检索可能忽略精确字符串匹配；另一方面，issue 描述或 Agent 查询中也经常包含行为描述、条件约束和测试现象，仅依赖关键词匹配又可能无法召回语义相关但字面不匹配的代码。因此，仓库级代码定位通常需要结合关键词匹配和语义相似度。此外，代码实体之间还存在包含、导入、调用、继承和重写等结构关系，相关代码未必与查询文本直接相似，而可能通过结构关系与入口节点相连。这也是本文进一步引入结构扩展的原因。

近年来，仓库级代码生成和补全研究已经开始关注检索增强在真实代码库中的作用。RepoCoder 通过迭代检索和生成利用仓库内相似上下文提升代码补全效果 \cite{repocoder}。CrossCodeEval 提出了更贴近仓库级跨文件依赖的代码补全评测，强调模型需要利用当前文件之外的上下文信息 \cite{crosscodeeval}。RLCoder 从强化学习角度优化仓库级代码补全过程中的检索策略 \cite{rlcoder}。EVOR 关注如何演化和改进检索到的知识，以提升代码生成质量 \cite{evor}。GraphCoder 进一步将代码上下文组织为图结构，并采用 coarse-to-fine 检索定位与补全目标相似的代码片段 \cite{graphcoder}。CodeRAG 则面向真实仓库级代码生成，同时构建需求图和 dependency-semantic code graph，以检索支持目标代码生成的相关代码 \cite{coderag}。GraphCodeAgent 在此基础上进一步将双图引导机制与 LLM Agent 结合，用于弥合自然语言需求和仓库实现之间的差距 \cite{graphcodeagent}。

这些工作共同说明，仓库级任务中的检索不能仅停留在短代码片段匹配层面，而需要结合项目内已有实现、跨文件依赖、接口约定和结构关系进行上下文构造。与上述工作相比，本文关注的任务不是代码补全或新代码生成，而是 SWE-bench 式缺陷修复 Agent 的多轮修复过程。因此，本文更强调检索结果如何作为 Agent 工具调用的一部分，被持续用于定位、深化、编辑和验证。

\subsection{面向仓库理解的代码图方法}

代码图通过节点和边表示程序实体及其关系，是建模仓库结构的重要方式。与上一节关注的“检索方法”不同，本节关注的是如何将代码仓库表示为结构化图，并利用图关系辅助模型理解项目。代码图中的节点可以表示文件、模块、类、函数、方法甚至语句，边可以表示包含、导入、调用、继承、数据依赖和控制依赖等关系。相比普通文本索引，代码图能够显式表示程序结构，使系统能够从一个候选节点进一步发现与其存在结构联系的相关代码区域。

RepoGraph 是近期代表性的仓库级代码图方法。该方法将仓库级结构作为可插拔模块，为 AI 软件工程系统提供 repository-wide navigation，并可接入不同的自动软件工程框架 \cite{repograph}。LocAgent 将代码库解析为包含文件、类、函数以及导入、调用、继承关系的异构图，使 LLM Agent 能够通过图上的多跳搜索执行代码定位 \cite{locagent}。CoSIL 也从软件 issue 定位角度出发，通过 LLM 驱动的代码仓库图搜索提升文件级定位效果 \cite{cosil}。RANGER 则构建较完整的仓库知识图，并将图节点描述和 embedding 结合起来，为仓库级检索 Agent 提供查询和遍历能力 \cite{ranger}。

在软件修复方向，KGCompass 构建 repository-aware knowledge graph，将 issue、pull request 与代码实体连接起来，并利用图路径辅助定位候选函数和生成补丁 \cite{kgcompass}。LingmaAgent 也强调通过仓库知识图和全局仓库探索帮助 Agent 理解项目结构，并结合搜索策略生成修复补丁 \cite{lingmaagent}。这些研究表明，结构化仓库表示能够缩小候选代码空间，并帮助模型发现 issue 描述中没有直接出现的相关代码位置。

图增强方法也被用于仓库级代码生成 Agent。CodeRAG 和 GraphCodeAgent 通过需求图和代码图建立自然语言需求与仓库实现之间的连接，使模型能够检索显式相关代码以及由依赖关系连接的隐式支持代码 \cite{coderag, graphcodeagent}。这类工作与本文具有较强相关性，因为它们同样关注如何让 LLM 在大型仓库中获得结构化上下文。不过，它们主要面向 repo-level code generation，而本文面向缺陷修复 Agent。缺陷修复任务不仅要求检索相关代码，还要求 Agent 在多轮过程中结合 issue、测试反馈、代码观察和修改尝试不断调整检索目标。因此，本文更关注图检索如何被组织成 Agent 可调用、可深化、可解释的工具。

现有图增强研究已经证明结构信息在仓库级软件工程任务中的价值，但与本文目标相比仍存在差异。第一，许多方法倾向于构建较完整的仓库图或知识图，而本文关注的是在计算成本受限的 SWE-bench 场景下，如何通过轻量骨架图与按需深化机制在全局覆盖和局部细节之间取得平衡。第二，部分方法强调一次性定位或固定流程修复，而本文更关注 Agent 在多轮工具调用过程中如何持续使用检索结果。第三，已有方法虽然已经开始将图检索作为 Agent 工具或上下文模块使用，但多数工作并不专门讨论检索结果如何面向 Agent 决策进行组织，例如如何同时呈现候选节点位置、方法摘要、相关原因、结构关系和置信度。本文的 Agent-facing context 设计正是对这一问题的进一步探索。

\subsection{研究空白与本文定位}

综上，现有研究可以大致分为三类。第一类是 SWE-agent、RepairAgent、AutoCodeRover 等面向软件修复或程序改进的 LLM Agent，它们强调工具调用、代码搜索、补丁生成和测试反馈，但对仓库结构化检索结果的组织仍有进一步改进空间 \cite{sweagent, repairagent, autocoderover}。第二类是 Agentless 等固定流程方法，它们通过定位、修复和验证等阶段降低自主 Agent 的复杂性，但相对缺少 Agent 多轮交互中的动态检索和深化能力 \cite{agentless}。第三类是 RepoGraph、LocAgent、KGCompass、GraphCoder、CodeRAG 和 GraphCodeAgent 等代码图或图增强检索方法，它们证明了仓库结构对于代码定位、代码补全、代码生成和软件修复的重要性，但并不完全等同于面向 mini-swe-agent 修复轨迹的轻量两阶段代码图检索框架 \cite{repograph, locagent, kgcompass, graphcoder, coderag, graphcodeagent}。

本文工作位于上述研究方向的交叉处。本文以 mini-swe-agent 这类轻量缺陷修复 Agent 为基础，引入代码图增强检索能力，使 Agent 能够在多轮修复过程中动态获取仓库级上下文。与原始 SWE-agent 主要关注 Agent-Computer Interface 不同，本文更关注如何为 Agent 提供结构化、轻量化、可交互的代码仓库表示。与固定流程方法不同，本文保留 Agent 的多轮工具调用能力，使其能够根据中间观察结果动态执行检索、深化和编辑。与已有代码图方法相比，本文采用两阶段代码图表示：首先构建轻量骨架图支持全仓库范围的快速定位，然后在 Agent 需要细粒度信息时，对候选文件或相关结构邻域执行按需深化，补充 method 级上下文。

此外，本文的检索设计强调多信号融合和面向 Agent 的结果组织。一方面，本文结合 BM25 关键词召回、语义向量检索和代码图结构扩展，使系统能够同时利用 issue 中的精确符号、Agent 查询中的语义线索以及程序结构关系。另一方面，本文通过 issue focus 模块从初始 issue 和后续 query 中抽取检索线索，并在多轮检索中保留逐渐衰减的初始 issue 信号，以减少 Agent 在后续局部探索中的检索跑偏问题。最后，本文将检索结果组织为面向 Agent 决策的结构化上下文，包括节点位置、方法摘要、相关原因、结构关系和置信度等信息，而不仅仅返回原始代码片段或文件列表。通过这种方式，本文希望在保持 Agent 交互灵活性的同时，降低其在真实仓库中构建代码理解的成本，并提升缺陷相关代码区域的定位效率。
